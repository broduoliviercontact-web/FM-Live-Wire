# ADR-0006 — État en mémoire, pas de DB (MVP)

- **Status:** Accepted
- **Date:** 2026-07-06
- **Supersedes:** none
- **Superseded by:** none
- **Liens :** PRD NFR-13, N-8 ; spine AD-6 ; addendum A.1 (invariant 8)

> **Immutability :** Accepted ADRs are immutable. Future changes must create a new ADR that supersedes this one.

## Contexte

Le MVP est une session radio live : un owner, 5–20 listeners, état volatile acceptable (un redémarrage = reconnexion requise, auto côté listeners). Il n'y a pas de replay, pas de presets, pas de persistance de séquences — tous hors MVP. Une DB ou Redis ajouterait une dépendance ops et un couplage que rien ne justifie à cette échelle.

Cependant, l'architecture doit rester **isolable** pour permettre un futur swap vers un Redis Streams adapter (scale-out multi-instance) **sans réécriture** des handlers.

## Décision

**État serveur en mémoire volatile, pas de DB, pas de Redis pour le MVP.**

- État en mémoire : `ownerPerformerId` (slot owner unique), `listeners: Map<socketId, …>`, `rateLimitBuckets: Map<socketId, …>`.
- Perte d'état au redémarrage acceptée (owner doit se reconnecter, listeners se reconnectent auto via Socket.IO).
- **Isolation** : `RelayService` derrière une interface d'adapter `{ broadcast(room, event): void }`. L'implémentation MVP est in-memory (`io.to(room).emit`). Le swap futur vers Redis Streams adapter se fait **sans toucher les handlers**.

## Conséquences

**Positives :**

- Aucune dépendance DB/Redis ; ops minimales ; démarrage instantané.
- Couplage MVP minimal ; swap futur préparé par l'interface d'adapter.
- Pas de TanStack Query (pas d'API REST métier) ; client state via Zustand.

**Négatives :**

- Pas de scale-out multi-instance MVP (état local par process).
- Pas de persistance (replay/presets) — non-objectif assumé.

## Alternatives considérées

- **Redis dès le MVP** : rejeté — over-engineering ; pas de scale-out ni de persistance requis. L'interface d'adapter prépare le swap.
- **SQLite/DB pour persistance** : rejeté — pas de besoin de persistance MVP (replay/presets hors scope).
- **Multi-instance + sticky sessions** : rejeté — mono-process largement suffisant.
