# ADR-0007 — Panic local côté listener

- **Status:** Accepted
- **Date:** 2026-07-06
- **Supersedes:** none
- **Superseded by:** none
- **Liens :** PRD FR-16, FR-17, FR-18, FR-24, S-2 ; spine AD-7, AD-17 ; recherche Correction 1

> **Immutability :** Accepted ADRs are immutable. Future changes must create a new ADR that supersedes this one.

## Contexte

Une note peut rester coincée (noteOff perdu sur un lien instable). Le listener doit pouvoir **couper le son sur sa sortie MIDI locale**, y compris **quand le serveur est injoignable** (kill backend, panne réseau). C'est la promesse de sécurité musicale du produit, testée par le critère de succès S-2 (« Panic local coupe les notes coincées, même serveur down »).

Une implémentation naïve aurait déclenché le Panic via un event serveur (`panic:trigger`) — mais cela casserait la promesse si le serveur est down, et violerait l'invariant one-way (un chemin retour listener→serveur→…). La recherche (Correction 1, 2026-07-06) verrouille : Panic = **purement local**, aucun event serveur.

## Décision

**Panic 100 % local côté listener, indépendant de l'état de connexion Socket.IO.**

- `features/listener/lib/panic.ts` ne dépend **que** de la `MIDIOutput` locale sélectionnée (réelle ou Mock) — **pas** de Socket.IO.
- **Panic standard** : CC 64 (sustain off) → CC 120 (all sound off) → CC 121 (reset controllers) → CC 123 (all notes off), `value 0`, × 16 canaux = **64 messages**, `send([0xB0|ch, cc, 0x00], performance.now())`. Action directe (geste fréquent, critique).
- **Force Panic** (opt-in, confirmation Dialog « Panic étendu : ~1–2 s ») : noteOff sweep 128 notes × 16 canaux = **2048 messages**. Dernier recours si le synthé ignore les channel mode messages.
- `PanicButton` sticky en bas du viewport, **jamais désactivé**, jamais masqué par un dialogue.
- **Aucun handler panic côté serveur** ; pas de télémétrie Panic agrégée MVP.
- Fail-safe musical (FR-24) : à la déconnexion listener ou perte de sortie, le scheduler arrête d'envoyer (pas de notes orphelines) ; reconnexion → reprise live sans replay.

## Conséquences

**Positives :**

- Panic fonctionne serveur down (S-2 satisfait).
- Respecte l'invariant one-way (aucun chemin retour pour le MIDI).
- Fail-safe musical : pas de notes orphelines.

**Négatives :**

- Pas de télémétrie Panic agrégée côté serveur — non-objectif MVP.
- Force Panic (2048 messages) peut prendre ~1–2 s — mitigé par avertissement + confirmation.

## Alternatives considérées

- **Panic via event serveur `panic:trigger`** : rejeté — casserait la promesse serveur-down + violerait one-way (Correction 1 verrouillée).
- **Panic automatique (sans geste)** : rejeté — un Panic inattendu casse la performance ; geste explicite requis.
- **Panic omni-canal (broadcast MIDI)** : rejeté — les channel mode messages doivent être envoyés **individuellement sur chaque canal** (pas de broadcast omni en MIDI) ; la boucle ×16 canaux est nécessaire.
